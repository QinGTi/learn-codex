Codex Skill 工作流

EPCC：让 Codex 做事更稳

EPCC 是一套适合复杂任务的 Skill（Skill，可复用任务能力/固定招式）：先探索，再计划，再编码，最后提交。它帮助我们把“快点做”变成“先想清楚，再一步步做对”。

Install EPCC

让 Codex 装好 EPCC

我们主要使用 Codex（Codex，能读写代码、理解项目、运行检查的 AI 编程助手）。所以安装 EPCC 时，可以让 Codex 先读取 GitHub 上的 EPCC 资料，再自动整理成 Codex Skill（Skill，可复用任务能力/固定招式）格式。

资料来源

先给 Codex 一个可靠来源

参考仓库： aws-solutions-library-samples/guidance-for-claude-code-with-amazon-bedrock

插件路径：assets/claude-code-plugins/plugins/epcc-workflow

这份资料原本是 Claude Code plugin，包含 12 个专门 agent、4 个 EPCC 命令和自动恢复 hook。课堂里不要求照搬全部工程配置，而是让 Codex 提取适合我们的 EPCC 工作流。

课堂说法

Codex 来做转换

学生不需要手动研究插件目录。我们只需要告诉 Codex：请阅读这个仓库，保留 Explore、Plan、Code、Commit 的核心规则，并生成一个可以放进本机 Codex skills 目录的 Skill。

复制安装 Prompt

把下面这段话发给 Codex，让 Codex 帮你完成读取、转换和安装。

请帮我安装 EPCC 工作流 Skill。 资料来源： https://github.com/aws-solutions-library-samples/guidance-for-claude-code-with-amazon-bedrock/tree/main/assets/claude-code-plugins/plugins/epcc-workflow 要求： 1. 先读取这个 GitHub 仓库里的 epcc-workflow 插件资料。 2. 它原本是 Claude Code plugin，请不要直接照搬插件格式。 3. 请提取适合 Codex 使用的 EPCC 核心规则：Explore、Plan、Code、Commit。 4. 请生成 Codex Skill 格式，安装到本机 Codex skills 目录。 5. 安装前如果目标目录已存在，请先告诉我差异，不要直接覆盖。 6. 安装后告诉我如何使用，例如“请使用 EPCC 工作流完成这个任务”。

观察 Codex 做了什么

看 Codex 是否先读资料、再整理规则、最后写入 Skill 文件。不要只看“装好了”，还要看它有没有说明来源和限制。

  • 是否保留 EPCC 四步流程
  • 是否说明这是从 Claude Code plugin 转换而来
  • 是否避免覆盖已有 Skill
  • 是否给出使用方法

用一句话启动

安装完成后，在复杂任务前加上这句话：

请使用 EPCC 工作流完成这个任务。

Codex 应该先进入 Explore 和 Plan，再开始 Code，最后根据需要做 Commit。

Lesson Question

为什么不能一上来就改代码？

常见问题

任务说得太快

如果只说“帮我做个页面”，Codex（Codex，能读写代码和运行检查的 AI 编程助手）可能不知道目标、风格、文件位置和怎样算完成。

EPCC 的答案

先把路标立好

EPCC 把任务拆成探索、计划、编码、提交四个阶段。每一步都有产物，老师和学生都能检查“现在做到哪了”。

一句话记忆：EPCC 不是让 Codex 变慢，而是让 Codex 少走弯路。

Key Concepts

先认识 4 个关键词

Skill

固定招式

Skill（Skill，可复用任务能力/固定招式）把一套常用步骤保存下来，下次遇到类似任务可以继续使用。

Spec

任务规格

Spec（Specification，规格说明）说明要做什么、给谁用、有什么限制、怎样算完成。

Task

小任务

Task（Task，任务）是可以单独完成和检查的一小步。EPCC 鼓励一次只推进一个清楚的小任务。

Acceptance

完成标准

Acceptance Criteria（Acceptance Criteria，完成标准）告诉我们：看到什么结果，就可以说这一小步完成了。

Workflow

EPCC 四步流程

Explore

先读项目和资料，理解现状。不要急着改文件。

  • 看目录和核心文档
  • 找已有风格和规则
  • 记录到 EPCC_EXPLORE.md

Plan

把目标变成规格和任务清单，让每一步都能被检查。

  • 写清输出物
  • 列出验收标准
  • 记录到 EPCC_SPEC.md 和 EPCC_TASKS.md

Code

按任务清单逐步实现，每完成一小步就验证。

  • 只改相关文件
  • 保留有用结构
  • 记录到 EPCC_CODE.md

Commit

确认质量后，只把相关修改提交成一个清楚的检查点。

  • 运行检查
  • 只暂存相关文件
  • 记录到 EPCC_COMMIT.md
Teacher Demo

老师如何演示

不推荐

一句话催 Codex 开工

帮我做一个 2048 游戏，直接写代码。

问题：没有要求先理解项目，也没有说明玩法、技术选择、测试方式和提交规则。Codex 可能很快动手，但老师和学生不容易判断它做得对不对。

推荐

用 EPCC 做 2048

请使用 EPCC 工作流，帮我从空仓库实现第一个网页版 2048 游戏。 第一步只做 Explore：检查仓库结构、现有规则和限制，写入 EPCC_EXPLORE.md。 第二步做 Plan：制定开发流程和测试流程，写入 EPCC_PLAN.md。 第三步做 Code：实现标准 2048，不是 2028；需要棋盘、移动、合并、计分、重开、best score 和移动端控制。 第四步做 Commit：运行测试，只暂存相关文件，提交到功能分支，并写入 EPCC_COMMIT.md。 完成标准：能在浏览器打开并游玩；核心逻辑有自动化测试；Codex 需要验证页面、说明测试结果，并且不要改无关文件。
演示观察点
  • Explore：Codex 是否先发现这是空仓库，并记录限制
  • Plan：是否把开发任务和测试验收拆清楚
  • Code：是否把 2048 逻辑和页面交互分开实现
  • Commit：是否测试通过后，只提交相关文件
老师追问
  • 如果一开始把 2048 说成 2028，Codex 应该直接猜吗？
  • 为什么移动合并规则要写自动化测试？
  • 为什么要用浏览器真的点按钮、按方向键检查？
  • 为什么提交前要建功能分支，而不是直接提交到 main？
Student Practice

学生模仿任务

任务

给自己的小工具做 EPCC 计划

选择一个学习生活小工具，例如单词卡、错题整理器、作息规划器。先不要写代码，只让 Codex 用 EPCC 帮你探索和计划。

请使用 EPCC 工作流，帮我规划一个小工具。 工具名称：____ 使用者：____ 解决的问题：____ 限制：只做一个 HTML 文件，适合初一学生使用。 完成标准：页面能打开，有输入区、结果区、检查清单，并且文字清楚。
替换练习

把空白换成自己的内容

  • 工具名称可以换成“英语单词卡”
  • 使用者可以换成“我和同桌”
  • 问题可以换成“复习单词容易忘”
  • 限制可以增加“不要联网，不保存隐私”

Verification

验收清单

我能用自己的话解释 Skill（Skill，可复用任务能力/固定招式）为什么适合保存工作流。

我能说出 EPCC 的四步：Explore、Plan、Code、Commit。

我知道复杂任务不能直接开改，要先理解项目和限制。

我能写出一个带目标、对象、限制和完成标准的 Codex Prompt（Prompt，提示词）。

我能检查 EPCC 记录文件，判断 Codex 有没有按步骤工作。

我知道危险操作、删除文件、提交代码之前都需要更谨慎地确认。

Reflection

复盘问题

问题 1

哪一步最容易被跳过？

想一想：你平时是不是最想直接进入 Code？为什么？

问题 2

谁负责判断？

Codex 可以执行任务，但目标、限制、安全和是否满意，仍然需要人来判断。

问题 3

怎样越用越省力？

当一套步骤反复有用时，就可以沉淀成 Skill，下次让 Codex 直接复用。
